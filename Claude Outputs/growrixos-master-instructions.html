Section 01 · Overview

What This Document Is

Think of this as the rulebook for the whole agency. Before anyone (or any AI) builds anything, they read this. It tells you what to build, how to plan it, how to design it, how to code it, and how to check if it's done right.

💡
Plain English Version: Imagine you're opening a restaurant. Before cooking, you need a menu, recipes, food safety rules, and a quality check. This document is all of that — but for building websites and digital products. Every product gets the same professional treatment, no shortcuts.
🎯
The Mission

Build high-quality HTML digital products that real businesses will buy — templates, tools, landing pages, and utilities — all planned and built with a proper system, not guesswork.

🤖
AI-First Approach

All planning, research, design specs, and code are produced with AI. Every output is written so that an AI can read it, understand it, and execute it — no ambiguity, no vague instructions.

Section 03 · The Build Pipeline

How Every Product Gets Built — 5 Phases

This is the most important part. No product gets built by jumping straight to code. Every single product goes through all 5 phases. Skipping phases = bad products = products nobody buys.

📚
Think of it like building a house: You don't just start laying bricks. First you plan, then you draw blueprints, then the architect checks everything, then you build, then inspectors check it. Same idea here.
Phase 01
🔍 Research
Who buys this? What do competitors look like? What price does it sell for?
Phase 02
📋 Planning
Define buyer, sections, features, content, and file structure.
Phase 03
🎨 Design Spec
Colors, fonts, layout wireframe, and the one unique signature element.
Phase 04
💻 Build
Write HTML, CSS, JS. Single file. Real copy. No lorem ipsum.
Phase 05
✅ QA Check
Mobile test, links, contrast, performance, and final review.
01
Research Phase — "Know Before You Build"
Before writing one line of code, answer these questions about the product you're about to build. This stops you from building the wrong thing.
  • Who is the buyer? (freelancer, small business, agency, creator?)
  • What problem does this product solve for that buyer?
  • What do the top 3 competitors on Gumroad, ThemeForest, or Envato look like?
  • What price range do similar products sell for? ($9? $29? $79?)
  • What's the primary action the user takes on this product? (Book a call? Download a file? Buy a product?)
  • What industry standard rules apply? (Email templates must work in Gmail/Outlook. Invoices must be printable.)
02
Planning Phase — "What Goes Inside"
Write a product brief. This is a simple document (can be in a comment at the top of the HTML file) that defines the product before any code is written.
📋 Product Brief Template (put at top of every file)
03
Design Spec Phase — "How It Looks"
Before writing CSS, define your design tokens. Think of tokens like the official color and font list — once set, everything must use them. No making up colors halfway through.
  • Pick a color palette: 1 background, 2–3 surface colors, 1 accent, 1 text color, 1 muted text color
  • Pick fonts: 1 display/heading font + 1 body font (from Google Fonts or system fonts only)
  • Decide the layout: Single column? Two column? Card grid?
  • Name the "Signature Element" — the ONE thing that makes this product memorable (e.g., a hover animation, a unique section layout, a bold typographic choice)
  • Decide if it's light or dark theme. Match the industry: finance = light/professional, music = dark/moody, fitness = bold/energetic
04
Build Phase — "Write the Code"
Now and only now do you write code. Always single-file HTML. Always real content. No generic filler.
  • Start with the <!-- PRODUCT BRIEF --> comment at the very top
  • Define all CSS variables in :root {} first before writing any other CSS
  • Write HTML structure first (all sections, no styling), then style it
  • Use real, realistic copy — fake business name, real phone format, real-looking prices
  • Mobile styles go inside @media (max-width: 640px)
  • No external JS libraries unless absolutely needed (use vanilla JS)
  • CDN links for fonts and icons only — FontAwesome, Google Fonts are fine
  • Every interactive element (button, input) must have a :hover and :focus state
05
QA Phase — "Does It Actually Work?"
Quality Assurance (QA) = checking if the product is ready to sell. Go through every item on the QA checklist (see Section 06) before marking a product done.
  • Open in browser and check all 3 breakpoints: mobile (375px), tablet (768px), desktop (1280px)
  • Click every button and link — they must work or be clearly marked as demo
  • Check that text is readable (dark text on light bg, light text on dark bg — WCAG AA contrast)
  • Check that fonts loaded correctly (no fallback sans-serif where a nice font was intended)
  • Check print layout for invoice, resume, proposal types
  • Validate: no broken HTML tags, no unclosed divs

Section 05 · Design Standards

Universal Design Rules — Always Apply

These rules apply to every single product, no exceptions. They are the baseline quality floor. If a product fails any of these, it doesn't ship.

Rule DO DON'T
Color Use a CSS variable system. 4–6 named colors max. One accent color that creates energy. Don't use more than 2 accent colors. Don't use pure #000000 black or #ffffff white as backgrounds.
Typography Pair a characterful display font (headings) with a clean body font. Import from Google Fonts. Set a clear size scale. Don't use 2 sans-serif fonts together (boring). Don't use more than 2 font families. Don't use system fonts for display text.
Spacing Use consistent spacing units (8px system: 8, 16, 24, 32, 48, 64). Sections should breathe. Don't stack sections with no breathing room. Don't add random padding values that break the rhythm.
Mobile Design mobile-first. Stack columns on small screens. Buttons min 44px tap target. Don't assume desktop. Don't use tiny text (below 12px) on mobile. Don't let horizontal scroll happen.
Content Write real copy that matches the buyer. Fake business names should sound real (e.g. "Meridian Consulting Group"). No lorem ipsum. No "Your Company Name Here." No "[Insert Testimonial]" placeholders.
Contrast Text must pass WCAG AA (4.5:1 ratio for body text, 3:1 for large text). Check with a contrast checker. Don't put gray text on gray backgrounds. Don't use colored text on colored backgrounds without checking contrast.
Animations Only animate where it helps: hover states, focus rings, modal opens. Make it subtle (150–300ms). No spinning logos. No bouncing text. No animations that run on loop automatically. No slide-in text effects that delay reading.
Images Use CSS gradients, SVG illustrations, or placeholder.co for demo images. Always add alt text. Don't hotlink random images. Don't use stock photos that have watermarks. Don't forget alt attributes.
Theme vs Industry Match theme to industry. Finance = light/trustworthy. Fitness = bold/dark. Legal = neutral/professional. Don't build a funeral home site in bright neon colors. Don't build a yoga studio in corporate gray.
Section 06 · Code Standards

How Code Must Be Written

Consistent code structure means any AI (or developer) can pick up any file and immediately understand it. Every file follows the exact same template structure.

📁 Standard File Structure (every HTML file must follow this order)
<!-- PRODUCT BRIEF: Name / Category / Buyer / Goal / Sections -->
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
<title>[Product Name] — Growrixos</title>
<!-- Google Fonts link -->
<!-- FontAwesome CDN if needed -->
<style>
/* 1. CSS Variables (:root) */
/* 2. Reset */
/* 3. Global / base styles */
/* 4. Layout (header, main, footer) */
/* 5. Components (cards, buttons, forms) */
/* 6. Sections (in page order) */
/* 7. Utilities */
/* 8. Responsive (@media queries) */
</style>
</head>
<body>
<!-- header -->
<!-- main sections in order -->
<!-- footer -->
<script>
/* All JS at bottom */
</script>
</body>
</html>
  • ✅
    Single File Only. All CSS and JS go inside the HTML file. The buyer downloads one .html file and it works — no folders, no dependencies.
  • ✅
    CSS Variables First. Every color and font must be defined in :root at the top of the style block. Never hardcode a hex color outside of :root.
  • ✅
    Semantic HTML. Use <header>, <main>, <section>, <article>, <footer>, <nav>. Never use <div> for everything.
  • ✅
    JS at Bottom. All JavaScript goes in a single <script> tag before </body>. No inline onclick="" attributes — use event listeners.
  • ✅
    No External Dependencies. Only Google Fonts and FontAwesome CDN are allowed. Everything else must be vanilla HTML/CSS/JS.
  • ✅
    LocalStorage for Stateful Tools. For trackers, checklists, and calculators, use localStorage to save user data between sessions.
  • ✅
    Print Styles. Invoices, résumés, and proposals must include a @media print block that hides navigation, shows clean content, and fits on A4 paper.

Section 09 · Product Spec

📧 Email Templates

Email templates have the strictest technical rules of all product types. Email clients (Gmail, Outlook, Apple Mail) render HTML very differently from browsers. One wrong CSS property can break the entire email.

⚠️
Critical Email Rendering Rules — Do Not Skip
These are non-negotiable — email breaks without them
▲
  • Use table-based layout — not flexbox or CSS Grid (Outlook doesn't support them)
  • Use inline CSS only — no external stylesheets, no <style> blocks (Gmail strips them)
  • Max width: 600px — anything wider breaks on mobile email clients
  • Use web-safe fonts as fallbacks: Arial, Georgia, Verdana (Google Fonts may not load in email)
  • Images must have width and height attributes set directly on the tag
  • All links must be full absolute URLs (https://...) not relative paths
  • Include a plain text <!-- [View in browser] --> link at the very top
  • Add unsubscribe link text at the footer (legal requirement in many countries)
  • Background colors go on the <td>, not the <table>
  • Buttons: use table cells styled as buttons, not <button> elements
⚠️ No Flexbox  ⚠️ No CSS Grid  ⚠️ No External CSS  ✅ Tables Only  ✅ Inline Styles  Max Width: 600px
Section 10 · Product Spec

📄 Documents & Reports

Invoices, proposals, and résumés need to look perfect on screen AND when printed. The buyer will literally print these and hand them to clients or employers. Print quality is not optional.

🖨️
Print & Document Rules
Must work on screen and on paper
▲
  • Include a @media print CSS block that hides nav, buttons, and decorative elements
  • Print layout must fit A4 paper (210mm × 297mm) — test this in browser Print Preview
  • Use light backgrounds for print — dark backgrounds waste ink and look bad printed
  • Invoice must auto-calculate totals with JavaScript (quantity × price = line total, sum = grand total)
  • Invoice must have: invoice number, date, due date, sender details, recipient details, line items, subtotal, tax, total
  • Proposal must have: cover section, problem statement, proposed solution, timeline, pricing table, terms, signature area
  • Résumé must be ATS-friendly: real text (not images of text), semantic headings, clean font hierarchy
  • A "Print / Download PDF" button must appear on screen but be hidden in print view
Theme: Always Light  Required: @media print  Price range: $19–$49
Section 11 · Product Spec

🛍️ E-Commerce & Marketing Pages

These are conversion machines. Every element exists to push the visitor toward one action — buy, sign up, or download. Distractions are the enemy.

🎯
Conversion & Marketing Rules
Every element must serve the CTA
▲
  • One primary CTA — don't confuse with 5 different buttons. Pick one action.
  • Hero must load above the fold with no scrolling. Headline + subheadline + CTA button + optional image.
  • Social proof (testimonials, logos, star ratings, user counts) must appear before the pricing section
  • Countdown timers must use JavaScript and count down in real-time
  • Product launch pages need: headline, product visual, benefits (not features), social proof, pricing, FAQ, final CTA
  • Link-in-bio pages: profile photo, name, bio, list of links as large tap-friendly buttons, social icons
  • App promo pages: screenshots in device frames, feature list, platform badges (App Store / Play Store), demo video slot
  • Coming soon pages: countdown timer, email capture form, social links
Rule: Single CTA focus  Required: Social proof section  Price range: $19–$59
Section 12 · Product Spec

🎓 Education & Info Products

Course pages, lead magnets, and webinar pages sell transformation. The buyer (student, reader, attendee) needs to believe their life will be better after. Trust, credibility, and specificity drive conversions here.

📚
Education Page Rules
Sell the transformation, not the information
▲
  • Course landing page sections in order: Hero → What You'll Learn → Who It's For → Curriculum → Instructor Bio → Testimonials → Pricing → FAQ → Final CTA
  • Lead magnet pages: clear statement of what the free resource IS, who it's for, what they'll learn, email capture form, sample preview
  • Webinar pages: event date/time (use JS to show "in X days"), speaker bios with photos, agenda/schedule, registration form
  • Portfolio templates: filterable project grid, about section, skills list, contact form — all in one scrolling page
  • Always show the instructor's face and credentials — anonymous courses don't convert
  • Pricing must show value: cross out "original price," show payment plan option if applicable
  • FAQ section must address the top 5 objections (Is this for beginners? How long does it take? What if I'm not satisfied?)
Required: Instructor bio with photo placeholder  Required: Curriculum/agenda section  Price range: $29–$79
Section 13 · Product Spec

🔧 Tools & Utilities

These are interactive HTML tools. The buyer wants something functional — they'll use it themselves or embed it. The product must actually work, not just look good.

⚙️
Tools & Utility Rules
Function first, then beauty
▲
  • Calculators: inputs must update output in real-time (no "Submit" button needed — use JS input event)
  • Calculators: show the formula being used so users understand how the number was calculated
  • Trackers and checklists: save state to localStorage so data persists when the page is refreshed
  • Trackers: include a "Reset / Clear All" button with a confirmation step (don't let users accidentally delete their data)
  • FAQ pages: use CSS-only accordion (no JS needed) — clicking a question expands the answer
  • Coming soon / maintenance pages: countdown timer with days/hours/minutes/seconds updating every second via JS
  • All tools must work offline (no API calls, no server required)
  • Tools must be embeddable — clean design that works inside any website's iframe
  • Add "How to use" instructions as collapsible section or tooltip system
Rule: Must work offline  Required: LocalStorage for stateful tools  Price range: $9–$39
