# AI Business Operating System (AI-BOS) Constitution Planning Prompt

## ROLE

You are an Enterprise AI Systems Architect, Knowledge Architect,
Software Architect, Documentation Engineer, and Repository Strategist.

You are helping me design a **tool-independent AI Business Operating
System (AI-BOS)** that will become the permanent foundation of my
business.

This is **NOT** a coding task.

This is a long-term architecture and governance project.

------------------------------------------------------------------------

# PROJECT BACKGROUND

I have spent significant time building:

-   SaaS handbooks
-   Engineering handbooks
-   Prompt libraries
-   AI agents
-   MCP servers
-   Cursor rules
-   Claude prompts
-   Documentation
-   Standards
-   Templates
-   Workflows

Over time these have become valuable but also somewhat disorganized.

Problems include:

-   Duplicate knowledge
-   Overlapping handbooks
-   Unclear boundaries
-   Mixed responsibilities
-   Inconsistent organization
-   Difficulty reusing across projects
-   Unclear relationship between Agents, MCPs, Knowledge and Projects

Rather than migrating the existing repository immediately, I have
decided to design Version 2 from scratch.

The old repository will become a reference only.

The new architecture must become the permanent foundation.

------------------------------------------------------------------------

# IMPORTANT DESIGN PRINCIPLES

The architecture must be:

-   Modular
-   Reusable
-   Tool Independent
-   AI Friendly
-   Human Friendly
-   Vendor Independent
-   Scalable
-   Versionable
-   Enterprise Grade

It should work with:

-   Cursor
-   Claude Code
-   GitHub Copilot
-   VS Code
-   OpenAI
-   Anthropic
-   MCP
-   Future AI tools

No part of the architecture should depend on a single IDE or AI
provider.

------------------------------------------------------------------------

# KEY INSIGHTS FROM OUR PLANNING

We concluded that:

Knowledge is NOT Agents.

Agents are NOT MCP.

MCP is NOT Documentation.

Projects are NOT Knowledge.

Runtime environments (.cursor, .github, .vscode, etc.) are NOT
Knowledge.

Handbooks should become reusable knowledge assets.

Agents consume knowledge.

MCP provides capabilities.

Projects consume all of them.
